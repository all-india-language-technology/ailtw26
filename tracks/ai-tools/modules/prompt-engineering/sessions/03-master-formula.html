C1 · Connection

A well-crafted prompt that still failed.

Read this prompt carefully. It has a persona, a guardrail, and a format instruction. By Chapter 2 standards, it is solid. But it produced useless output. Why?

A well-built prompt
Act as a senior translation consultant. Review the current draft of Mark 5:1–5 for back-translation accuracy issues. Flag any divergences from the ESV source text. Do not speculate. Flag uncertain items as NEEDS VERIFICATION. Return as: Verse | Divergence | Severity (Major / Minor / Flag).
❌
The AI's actual response: "I'd be happy to help review Mark 5:1–5 for back-translation accuracy. However, I don't see any draft text in our conversation. Could you please paste the draft and the ESV source text so I can compare them?"
💬

Discuss before we explain

  1. What is missing from this setup that no building block could fix?
  2. Could you write a "perfect" prompt for this task without having the draft text available to paste in?

The prompt is technically correct. The setup is incomplete. The prompt is only one of four things the AI needs to do its job. Chapter 3 maps the other three.
C2 · Content

The four components of strong AI output

Think of this as the full briefing you would give an expert colleague on their first day. The prompt is the instruction — but without documents, access, and standing references, even the best instruction produces guesswork.

Strong AI Output =

[Instruction] + [Materials] + [Tools] + [Additional Info]

1 Component 1 · Instruction ›

What it is: The prompt itself — the role, task, constraints, and output format you craft using the techniques in Chapter 2.

The most common missing element: Most practitioners have the task right but omit the format. Specifying output format alone — adding "return as a table" or "return as a numbered checklist" — often doubles the immediate usability of the output.

Weak instruction
Explain Mark 5.
Strong instruction
Act as a translation consultant. Analyse Mark 5:1–20 for a rural translation team in a shame-honour culture. Do not speculate beyond the text. Return a table: Verse | Cultural Insight | Translation Risk.
2 Component 2 · Materials ›

What it is: The material the model can actually see and reason over — pasted text, uploaded files, connected project sources. If the relevant text is not present in the conversation, the model fills the gap with statistical guesses.

Rule of thumb: No source text in the materials = no trustworthy translation output. The model cannot reason accurately over material it cannot see.

SourceHow to supply it
Current draft passagePaste into the chat
Back-translationPaste alongside the draft, clearly labelled
ESV / ULT source textPaste in, labelled "SOURCE TEXT:"
Team checking notesPaste or upload as a text file
Key-term decisionsPaste or upload your key-terms CSV
Commentary excerptPaste the relevant section with attribution
3 Component 3 · Tools ›

What it is: Registered capabilities the AI can call — operations that go beyond generating text: web search, file reading, calculations, and in advanced setups, direct connections to external systems.

Today — Standard chat

In a standard chat window (Claude, ChatGPT, Gemini), AI tools are limited to what it can do with text you supply. You paste; the AI reads, searches, compares, transforms. Powerful but manual — you are the bridge between the AI and your project files.

Emerging — Connected environments

When tools are registered via MCP or similar integration, the model gains direct access to external systems. It can fetch data directly, call specialist tools (AQuA, PTXprint, Whisper), and chain multiple operations in a single prompt.

🔭
Imagine a Copilot for Paratext: One prompt could fetch the current draft from Paratext, submit it to AQuA for quality scoring, generate checking questions for verses below threshold, and return a table — without the user touching a file. This is the direction the field is heading.
🔒
Data safety: Offline/local tools (Whisper via LM Studio, PTXprint, XeLaTeX) are safe for unpublished draft text — no data leaves the device. Cloud tools should only receive text covered by your organisation's data agreements. See Chapter 4, §4.5 for the full decision rule.
4 Component 4 · Additional Info (Standing References) ›

What it is: Persistent reference documents uploaded or pasted at the start of each new chat — your stable memory layer. Without these, you repeat the same project briefing in every session. Each new chat starts from zero.

DocumentContents
ai-session-brief.mdProject context, target audience, theological guardrails, standing constraints, key decisions
key-terms.csvApproved renderings for key theological and cultural terms
style-guide.mdTone, register, reading level, what to avoid
decision-log.mdChecking-round decisions that should not be revisited

How to use it: At the start of every new chat session, paste or upload these documents before your first prompt. They set the frame for the entire session.

The Iteration Loop: Conversational Steering

A strong first prompt does not always produce a perfect first output. This is normal and expected. The skill of conversational steering — building on the previous response rather than starting over — multiplies the value of every prompt you write.

  • 1
    Initial ask — get the structure right "Explain the significance of the pigs in Mark 5 for a translation team."
    Result: well-structured but too academic.
  • 2
    Refinement — fix what's wrong "Good start. Now remove the jargon and rewrite in plain language — 3 sentences maximum."
    Result: simpler, but still prose when you need a table.
  • 3
    Final transformation — get the format right "Now format that as a 2-column table: Verse Range | Cultural Insight."
    Result: ready to paste into your checking notes.
    • SituationSteering instruction
    Output is too long"Good. Now reduce this to 3 bullet points maximum."
    Output is too generic"This is too general. Add specific examples from Mark 5:1–20."
    Output used wrong vocabulary"Replace all theological jargon with plain English equivalents."
    Output speculated too freely"Remove any speculation. Only state what is supported by the text itself."
    Output used the wrong format"Reformat this as a table: Insight | Verse Reference | Risk."
    C3 · Challenge

    Diagnose the broken setup

    Three AI session setups. Each is missing at least one component of the Master Formula. Identify which component is absent — before seeing the analysis.

    Setup A

    The session: A translator opens a new chat, types a careful prompt asking the AI to help plan the FIA for Mark 5, and starts working. The AI produces a detailed FIA plan — but it references cultural features from a completely different cultural context, and the suggested resources don't exist in their language.
    🤔

    Which component is missing?

    The instruction (prompt) was solid. What else was absent from this setup? What would you tell this translator to prepare before their next session?

    📋
    Missing: Additional Info (Component 4). The translator started from zero — no AI session brief, no key-term decisions, no knowledge of the project's target audience or language situation. The AI defaulted to generic assumptions about culture and resources. Pasting an AI session brief (project context, target community, language name, relevant cultural anchors) at the start of every session would have corrected this.

    Setup B

    The session: A consultant has a detailed AI session brief loaded and a strong prompt ready. They ask the AI to review the back-translation of Mark 5:1–5 for implicit information gaps — but they forget to paste the back-translation. The AI produces a generic analysis of "common back-translation gaps in narrative passages."
    🤔

    Which component is missing?

    The instruction was strong. The standing references were loaded. What did the consultant forget?

    📄
    Missing: Materials (Component 2). Without the actual back-translation text in the conversation, the AI had no real text to reason over. It produced a generic template rather than a focused analysis. The fix is simple: always paste the relevant passage text before writing the analytical prompt. No source text in the materials = no trustworthy translation output.
    👥 Cross-lane pair activity

    Setup C

    The session: A cluster manager wants to use AI to help triage a backlog of 60 community testing reports — identifying which ones flag significant accuracy issues vs minor naturalness comments. The AI is excellent at reading and summarising, but the reports are in three different file formats (PDF, Word, hand-typed notes) and the manager has to manually copy-paste each one into the AI, one at a time. The session takes four hours to complete something that could take thirty minutes.
    🤔

    Discuss with a cross-lane partner

    Lane 3 person: What tool setup would have made this faster? Lane 1 or 2 person: what would you want to know about these reports before the AI touched them? What are the governance questions?

    🔧
    Missing: Tools (Component 3) and — at a deeper level — system design. The manager was manually bridging between files and the AI, doing work that a connected environment (batch file upload, structured extraction tool) would handle automatically. The stronger session design: upload all reports as text files in one batch, use a structured extraction prompt with a classification rubric, and return a triage table. The governance question the cross-lane partner should have raised: are these community testing reports considered unpublished data? If so, which tool tier is appropriate?
